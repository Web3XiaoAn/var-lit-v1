# Var-Lit V1：整体架构、核心逻辑与模型由来

本文描述当前公开仓库的实际实现。它不是收益承诺，也不是面向所有市场条件的通用模型。
当前可交易资产被启动检查固定为 BTC；XAU 与代币化原油只保留未来按资产独立校准的结构，
尚未启用。

## 1. 系统边界

```text
Variational 网页 + OKX Wallet
          │
          │ Chrome Bridge：行情 WS / REST 观察 / 命令回执
          ▼
┌──────────────────────────────────────────────────────────────┐
│ Python Runtime                                               │
│  行情归一化 → 三窗统计 → 开仓门槛 → 执行存活校验             │
│       → Firm Quote/Commit → Lighter IOC → 持仓/平仓/恢复      │
│       → 权威仓位与活动订单对账 → 原子恢复状态与执行证据       │
└───────────────┬──────────────────────┬───────────────────────┘
                │                      │
                ▼                      ▼
       Lighter SDK / WS        127.0.0.1:8780 运维面板
                                       │
                               SSH 本地端口转发查看

轮转执行证据 ──低优先级独立进程──► SQLite 研究库 ──► 只读分析器
```

三条 Chrome Bridge 通道只绑定服务器回环地址。运维面板也只监听回环地址，公网防火墙不应
开放其端口。Runtime、Chrome、显示服务和研究采集器由不同 systemd 服务管理，停止终端或
VNC 不会等同于停止 Runtime。

## 2. 一轮交易的状态机

### 开仓候选

1. 同步获取 Variational 与 Lighter 的可成交价格，拒绝过期、缺档或序列中断的数据。
2. 分方向计算 5m、30m、1h 经验分布；只有三个窗口都就绪才产生候选。
3. 以三窗最高中位数为基座，并用中位数至 Q80 的加权跨度内插 Q58 动态门槛。
4. 500U 参考价和当前配置金额的 Variational Firm Quote 都必须越过同一冻结门槛。
5. MAD 尖峰保护、最近 5 秒振幅和 `execution-survival-v2` 只否决当前机会；不会固定抬门槛，
   也不会因普通盈亏暂停自动化。

### 双腿执行

1. Runtime 请求 Variational Firm Quote，核对金额、方向、时效和经济价格。
2. Commit 被接受后，必须继续通过账户快照或成交证据确认真实结果；HTTP 200 或 `rfq_id`
   本身不等于成交。
3. Lighter 对冲腿使用确定性 client order id、完整盘口、经济限价和 Market IOC。
4. 开仓 IOC 最多进行受保护重试；若只剩少量未对冲数量，则在仍有新鲜完整盘口且不超过
   IOC 滑点上限时执行一次 `open-hedge-recovery-v1`。仍失败才回退 Variational，避免长期
   单边敞口。
5. 每次 Quote、Commit、Var fill、Lighter ack/fill 和恢复动作都写入可重放执行证据。

### 持仓与平仓

- 冻结本轮实际成交数量、方向、开仓价格、门槛和开仓金额；后续参数刷新不能重写历史。
- 30 分钟前只有整轮估值为正，并满足代码中的零磨损持续条件，才允许提前平仓。
- 满 30 分钟后使用配置的基础整轮磨损下限；满 1 小时后按实际开仓金额等比例使用超时
  阶梯下限。新一轮恢复基础下限。
- 平仓 Lighter 腿先按最新完整盘口进行受保护 reduce-only IOC；保护重试失败后进入既有
  reduce-only 恢复路径。订单结果不明、未知活动订单或真实账户不一致仍属于安全异常。

### 对账与恢复

Runtime 以 Variational 权威账户快照、Lighter 私有流/账户接口、活动订单和本地未决意图
共同判定状态。普通亏损、突破磨损线或单次机会被拒绝不会自动停机；只有无法安全判断的
单边敞口、仓位不一致、未知活动订单或订单结果不明会触发安全暂停。持仓期间暂停新开仓
不会关闭既有仓位的平仓、恢复和对账。

## 3. `adaptive-median-v6` 的由来

它是公开 V1 内部的策略状态版本，不代表公开发布过六个产品版本。模型文件记录了校准
数据哈希、资产、参考金额、覆盖时长、公式和分方向窗口统计，便于复现与审计。

现行公式为：

```text
跨度 = 0.25 × (5m Q80 - 5m median)
     + 0.45 × (30m Q80 - 30m median)
     + 0.30 × (1h Q80 - 1h median)

门槛 = max(5m median, 30m median, 1h median)
     + (58 - 50) / (80 - 50) × 跨度
```

设计目的不是追逐一瞬间的极端基差，而是要求机会先高于三个时间尺度的常态，再让当前
分布宽度决定额外余量。5m/30m/1h 权重分别为 25%/45%/30%；Q58 是冻结在模型文件中的
经验分位。MAD 与 5 秒振幅是机会级保护，不修改概率门槛本体。

仓库中的校准文件明确标记 BTC、500U 参考金额和约 4.10 小时初始覆盖。它只用于确定
公式初值；运行时的三窗中位数和分位数持续由当前行情更新。因此这个模型仍须通过更长
时间、工作日/周末和不同波动状态的前向样本继续验证。

## 4. `execution-survival-v2` 的由来

三窗门槛回答“基差是否有统计余量”，但不能回答“该余量能否活过随机执行链”。第二层
模型以约 450ms 的目标执行窗口估计当前机会的存活概率，按 BUY/SELL 分开校准：

- 门槛以上的余量（bps）；
- 最近 5 秒方向振幅；
- 1bps 范围内方向归一化深度不平衡；
- 1000ms 方向 book flow；
- 100ms 方向 trade flow；
- 方向归一化 microprice 偏移。

样本严格按时间顺序以前 80% 拟合、后 20% 校准，没有随机打乱未来数据。冻结目标存活率
为 92%；模型文件同时记录每个方向的训练数、校准数、系数、门控值和因子下限。它是简单、
可解释的机会级门控，不是复杂黑箱预测：失败只跳过这一笔，下一笔报价继续判断。

当前模型只含 BTC 分方向参数。未来如果加入 XAU 或代币化原油，应使用相同的 bps、深度
占比和方向归一化特征接口，但为每个资产、方向单独校准；不能直接复制 BTC 的绝对参数。

## 5. 数据、研究与版本纪律

- Runtime 热路径只使用内存和原子状态文件，不直接写长期 SQLite。
- 独立研究服务按内容哈希去重保存背景、原门槛候选、动态候选和实盘样本，并区分
  `observe/live`、资产、方向、策略版本和样本类型。
- 统一只读分析器汇总随机延迟链、100/250/450/1000ms 存活、reserve、深度、book/trade
  flow、microprice、保护回退、方向和完整轮次收益。
- 模型升级应按时间划分训练、验证和最终测试；旧实盘样本保留版本标记，但生产代码只保留
  当前策略实现。
- TFI/OFI、多档深度、跨市场领先滞后以及在线覆盖率/分布漂移校准属于后续研究方向，
  未在当前模型中伪装成已经验证的能力。

## 6. 运维与安全原则

- `.env`、`runtime.env`、Chrome profile、钱包、API 私钥、数据库、日志和运行状态永不进入 Git。
- observe → live 必须人工修改配置并重启，程序不会自行升级权限。
- 面板按钮采用权威快照预览与 60 秒二次确认；变化后的旧确认不可复用。
- 刷新 Variational 页面是最高优先级的恢复动作之一，但刷新本身不证明订单未成交；刷新后
  仍须重新读取权威账户、活动订单与成交记录。
- 任何强制平仓、残仓处理或恢复动作都应先核对两边账户，不能只相信本地缓存。
