# adaptive-median-v3 三窗口中位数入场

v3 保留既有 Variational Firm Quote/Commit、最新 Lighter 深度复核、200U开仓硬上限、精确数量平仓、账户对账、异常恢复和单轮canary。Chrome命令协议没有变化。

## 中位数与方向门槛

每个方向都先把套利收益率标准化为“数值越大越有利”。5m、30m和1h窗口使用中位数，避免短暂尖峰、陈旧报价或薄深度异常值直接拖动基准。方向 `d`：

```text
Center_d = max(Median_5m_d, Median_30m_d, Median_1h_d)
Margin_d = max(0.50bps, 0.25 × MAD_30m_d)
Entry_d  = Center_d + Margin_d
Final_d  = max(Entry_d, Balance_d)
```

因此 `Final_d` 必然高于三个窗口中位数。方向均衡线只会在单侧机会片段超过另一侧两倍时提高过度活跃方向，不会降低基础门槛。

500U参考深度先以冻结门槛产生候选；Variational Firm Quote返回后，再用实际Firm数量对应的200U Lighter VWAP检查同一门槛。成本线与加权Q95退出估计继续写入面板、冻结上下文和日志，但v3不让预测成本线否决已经通过三窗口门槛的机会。报价新鲜度、跨源时差、账户一致性、活动委托和金额上限仍是硬保护。本地命令通道不再要求人工共享密钥，仍固定为loopback并校验Chrome扩展来源、协议与构建版本。

## 刷新与延迟

行情/可执行价差沿用事件驱动刷新，统计样本每秒一次。5m、30m和1h都按当前时刻向前滚动，旧数据逐秒被新数据替换。参数候选每5分钟在后台线程计算；第一次完整1h窗口立即生效，之后每个合格候选也立即激活，不再因两次确认造成10分钟延迟。在途订单只读取自己已经冻结的epoch并做常数次比较，不增加网络等待。内存统计只使用最新一小时；JSONL至少保留61分钟，每9分钟在后台原子压缩一次，所以磁盘跨度最多约70分钟。

## 平仓

v3沿用v2精确整轮底线，不把中位数回归作为第二个硬条件：

```text
floor(t) = 0U,                         t < 30m
         = -1.0bps × open_notional,    t >= 30m
```

只有实际开仓收益、当前平仓收益与平仓reserve组成的整轮下界达到对应floor时才平仓。120分钟只告警，不强制突破底线。

## 本轮回放依据

本次canary2使用最近一小时有效实盘样本回放余量。按15秒合并机会片段，旧 `1.0bps` 余量得到BUY 0、SELL 4，方向均衡后SELL仅保留2；新 `0.50bps` 余量得到BUY 36、SELL 53，当前分布无需额外抬高均衡线。这里统计的是门槛机会片段，不是预计成交次数或收益预测；正式交易仍需通过Firm复核、行情新鲜度、账户一致性和单轮canary限制。
